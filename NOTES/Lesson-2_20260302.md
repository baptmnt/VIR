# Lecture 2 – Container Orchestration \& Cloud Architecture

**Objective:** Master the “brain” that manages distributed containers and specialized Telco workloads.

***

## Introduction (10 min)

### The "Day 2" Problem: managing hundreds of containers

- **Day 0 / Day 1 / Day 2 distinction**
    - Day 0: design/architecture of the platform (what cluster, regions, network layout).
    - Day 1: initial deployment of the cluster and first applications.
    - Day 2: ongoing lifecycle—upgrades, failures, scaling, security hardening, debugging complex issues.[^1][^2]
- **Why “just Docker” isn’t enough**
    - Running isolated containers on a single host solves packaging and dependency issues but not fleet management.
    - Questions: where to place new containers, how to restart them on failure, how to expose them on the network, how to roll out updates with zero downtime.
- **Kubernetes as the "container control plane"**
    - Kubernetes provides a **declarative API**: you describe desired state (YAML), and controllers continuously drive the actual state toward it.[^3][^2]
    - Handles scheduling, health checks, auto-restarts, scaling, and service discovery across many nodes.
- **Telco framing: from manual NMS to programmable control planes**
    - Traditional telco NMS/OSS tools orchestrate network elements with proprietary interfaces.
    - Kubernetes plays a similar role for CNFs and microservices-based telco functions, but via open APIs and cloud-native patterns.

## Kubernetes (K8s) Fundamentals (30 min)

### Internal Mechanics: Control Plane \& Worker Nodes

- **Cluster high-level structure**
    - **Control Plane**: decides “what should happen” (API server, scheduler, controllers, etcd).[^2][^3][^1]
    - **Worker Nodes**: run the actual application containers (kubelet, kube-proxy, container runtime).
- **kube-apiserver – the front door**
    - HTTP/JSON API that all tools use: `kubectl`, controllers, operators, dashboards.[^4][^3][^2]
    - Validates requests, enforces authentication/authorization (RBAC), and persists or retrieves objects via etcd.
- **etcd – the source of truth**
    - Distributed **key-value store** that holds the entire cluster state (Pods, Deployments, ConfigMaps, etc.).[^3][^2]
    - Only the API server talks directly to etcd; all changes to the cluster go via the API.
- **kube-scheduler – where to place Pods**
    - Watches for **unscheduled Pods** (no node assigned) and selects an appropriate node based on resource requests, taints/tolerations, affinity rules, and more.[^5][^4][^3]
    - Produces a **binding**: Pod X → Node Y, after which kubelet on that node will create the containers.
- **kube-controller-manager – reconciliation loops**
    - Runs many controllers (Deployment, ReplicaSet, Node, Service, etc.).[^5][^4][^3]
    - Each controller continuously compares **desired state** in etcd (e.g., 3 replicas) with **actual state** and takes corrective actions (create/delete Pods, update endpoints).
- **Node components: kubelet \& kube-proxy**
    - **kubelet**: agent on each worker node that watches for Pods assigned to that node via the API, then talks to the container runtime (containerd/CRI-O) to start/stop containers and reports their status.[^6][^7][^3]
    - **kube-proxy**: implements Service networking (often via iptables or IPVS) so that traffic sent to a Service IP is load-balanced to the right Pods.[^7][^6][^3]
- **Under the hood: what happens when you apply a Deployment YAML**
    - API server writes Deployment object to etcd.
    - Deployment controller creates a ReplicaSet; ReplicaSet controller creates Pods.
    - Scheduler assigns Pods to nodes; kubelet pulls images and runs containers; kube-proxy updates routing for Services pointing to those Pods.

***

### Primitives: Pods, Services, Deployments

- **Pods – smallest deployable unit**
    - A Pod wraps one or more tightly coupled containers that **share the same network namespace and storage volumes**.[^8]
    - All containers in a Pod share the same IP and ports; they can talk over `localhost`, which simplifies patterns like sidecars and init containers.[^8]
- **Why Kubernetes has Pods, not “just containers”**
    - Pod = “logical host” abstraction: containers inside behave as processes on the same machine.[^8]
    - Enables sidecar patterns (e.g., logging agent, metrics collector, envoy proxy) attached to the main application container.
- **Services – stable virtual IPs and discovery**
    - Service groups a set of Pods (selected by labels) behind a **stable DNS name and virtual IP**.[^9][^10]
    - Types:
        - ClusterIP: internal-only virtual IP inside the cluster.
        - NodePort: opens a port on all nodes to expose the Service externally.
        - LoadBalancer: asks cloud provider to provision an external LB pointing to the Service.[^10]
- **Deployments – declarative stateless controllers**
    - A Deployment declares the desired number of **replicas** of a Pod template and manages rollouts/rollbacks.[^11][^8]
    - Rolling updates: creates new Pods, gradually shifts traffic, then deletes old ones while respecting availability constraints.[^11]
- **Under the hood: traffic flow example**
    - Client → LB (cloud) → NodePort → kube-proxy → selected Pod IP.
    - When a Pod dies, the controller creates a new Pod, and the Service’s endpoints list is updated automatically; clients keep using the same Service IP.[^9][^10][^8]

## Modern Cloud \& Multi-Cluster (20 min)

### VPC \& Regions: designing for HA and DR

- **Virtual Private Cloud (VPC) basics**
    - Logical, isolated network in a cloud provider with its own subnets, routing tables, and security rules.[^12][^13]
    - Kubernetes nodes (control plane + workers) live in subnets; Services with LoadBalancer type get VPC-aware load balancers.
- **Regions and Availability Zones (AZs)**
    - Region = geographic location; AZ = physically separate datacenters within a region.[^14][^15][^12]
    - Multi-AZ clusters spread worker nodes (and often control plane) across AZs so that a zone outage doesn’t kill the entire cluster.
- **Kubernetes across multiple zones**
    - One cluster can span multiple zones; scheduler places Pods across zones; if a zone fails, Pods are rescheduled in remaining zones.[^15][^12]
    - Storage and network components must support multi-AZ (e.g., replicated storage, zonal-aware load balancing).
- **Disaster Recovery (DR) strategies**
    - **Intra-region HA:** multi-AZ cluster with zonal spread.
    - **Cross-region DR:** multiple clusters in different regions, global DNS or load balancer to failover between them.[^13][^16][^12]
    - Data layer often uses replication (e.g., multi-region storage, database replication) to ensure RPO/RTO objectives.
- **Telco-specific angle**
    - 5G core or IMS clusters may be deployed across AZs for high availability and across regions or countries for regulatory and latency reasons.
    - Control planes for CNFs must be aware of data-plane locality (UPFs near base stations; control-plane components in regional hubs).

***

### Multi-Cluster Strategy: segmentation \& federation for Data Sovereignty

- **Why multiple clusters?**
    - Isolation of environments (prod vs staging), tenants, or high-risk workloads.
    - Scaling beyond one cluster’s limits; resilience to full-cluster or region failures.[^17][^14][^12]
- **Data sovereignty and regulatory constraints**
    - Certain subscriber or network data cannot leave a country or region (e.g., GDPR, local telecom regulations).[^18][^19]
    - Operators run clusters in each jurisdiction; data stays local, while control and observability may be federated.
- **Multi-cluster connectivity \& traffic management**
    - Global load balancers or DNS-based routing distribute traffic across clusters; each cluster exposes Ingress or gateways.[^20][^12][^15]
    - Service meshes or multi-cluster gateways route service-to-service traffic securely across clusters and regions.[^18][^20]
- **Config and policy distribution**
    - GitOps: same Helm charts or Kustomize overlays applied across multiple clusters with environment-specific overrides.
    - Central policy engines (e.g., OPA/Gatekeeper) and fleet-management solutions push common security and network policies.
- **Telco examples**
    - Separate clusters per country or per big enterprise customer, with central monitoring and limited cross-cluster signaling.[^21][^18]
    - Edge clusters close to RAN or enterprise sites, plus central clusters for core services and analytics.

## Stateful vs. Stateless (10 min)

### Managing persistent storage: PersistentVolumes vs disposable apps

- **Stateless applications**
    - Do not persist critical data locally; any instance can handle any request (e.g., API gateways, web frontends).[^22][^23]
    - Easy to scale horizontally by increasing replica count; failures are cheap.
- **Stateful applications**
    - Maintain data across requests and restarts: databases, message queues, subscriber state stores, some network functions.[^22]
    - Need persistent storage, stable identities, and often ordered startup/shutdown.
- **Kubernetes storage primitives**
    - **PersistentVolume (PV):** abstract storage resource in the cluster (backed by cloud volumes, NFS, Ceph, etc.).
    - **PersistentVolumeClaim (PVC):** a Pod’s request for storage; bound to a PV that satisfies its requirements (size, access mode).[^24][^22]
    - StorageClasses automate dynamic provisioning of PVs when PVCs are created.[^16]
- **Controllers: Deployment vs StatefulSet**
    - Deployment: designed for stateless Pods; replicas interchangeable; Pods get random names; may share the same PVC, if any.[^25][^23][^24]
    - **StatefulSet:** stable Pod identities (pod-0, pod-1, …) with dedicated PVCs via `volumeClaimTemplates`; used for databases, distributed systems.[^23][^24]
- **Telco state examples**
    - Subscriber profiles, session state, billing records, CDRs, and some control-plane elements require persistence.
    - Often implemented via CNFs that rely on Kubernetes PVs/StatefulSets under the hood, or external managed databases.

## Edge Computing \& Autonomy (20 min)

### Ultra-Low Latency: processing near IoT or base stations

- **Edge computing motivations**
    - Reduce latency for URLLC and time-sensitive services (industrial control, AR/VR, vehicular communications).
    - Save bandwidth and backhaul costs by processing data locally before sending aggregates to central cloud.[^26][^27]
- **Constraints at the edge**
    - Limited CPU, memory, and storage; sometimes ARM-based hardware.[^27][^26]
    - Intermittent or constrained connectivity to central datacenters.
- **Lightweight Kubernetes distributions**
    - **K3s, MicroK8s**: CNCF-certified but optimized for resource-constrained environments, fewer dependencies, smaller footprint.[^26][^27][^21]
    - Still expose the standard Kubernetes API, so CNFs and management tools can work with minimal adjustments.
- **Networking at the edge**
    - Need efficient data-plane integration with 5G UPF or gNB side; often rely on CNI plugins with SR-IOV, DPDK, or eBPF-based acceleration.[^27]
    - Telco edge nodes may host CNFs that handle user-plane traffic with stringent performance requirements.
- **Under the hood: scheduling and failure in a constrained environment**
    - Scheduler must respect node capacities and latency constraints; fewer nodes means less flexibility.
    - Pod anti-affinity and topology spread constraints help avoid placing critical replicas on the same fragile node.

***

### Edge Autonomy: local operations during central disconnection

- **Autonomy requirement**
    - Edge sites must continue operating critical services even if the link to the central cloud is degraded or down.
    - Particularly important for emergency services, industrial plants, and remote sites.
- **Local control planes vs centralized control**
    - Fully independent clusters at the edge: each with its own control plane; resilient to WAN failures.
    - Alternatively, centralized control plane with local worker nodes—simpler but more vulnerable if control-plane connectivity is lost.
- **Workload design for autonomy**
    - Deploy local versions of necessary control-plane services (e.g., local caches, config stores, monitoring agents).
    - Use patterns where data can be buffered locally and synced later when connectivity returns.
- **Kubernetes mechanisms enabling autonomy**
    - Node-level caching and kubelet’s ability to run existing Pods even if temporarily disconnected from API server.
    - Local operators that manage CNFs at the edge, with occasional synchronization with central GitOps/management tools.[^21][^26]
- **Telco examples**
    - Local 5G core slices or UPFs at base-station clusters that keep routing user traffic within a campus even if central OSS/BSS or core cluster is unreachable.
    - Edge NFs that maintain local policy/charging decisions and sync CDRs later.

## GPU Orchestration for the AI Era (20 min)

### Specialized Hardware: GPUs and DPUs in Kubernetes

- **Why GPUs/DPUs in telco clusters**
    - AI for network optimization, fraud detection, anomaly detection, RAN optimization.
    - GPU-accelerated media processing, transcoding, and inference in edge video applications.
- **Kubernetes device plugins**
    - Vendors provide **device plugins** that register specialized resources (e.g., `nvidia.com/gpu`) with the kubelet.[^2][^7]
    - Scheduler can then assign Pods requesting GPUs to nodes where these resources are available.
- **Requesting and limiting GPUs**
    - Pods specify GPU resources in `resources.requests/limits`; scheduler treats them as allocatable like CPU/Memory.
    - Isolation is handled by device drivers and runtime hooks; containers get access to specific GPU devices.
- **DPUs / SmartNICs integration**
    - Offload networking, encryption, or storage tasks from CPU to DPUs, often via SR-IOV or vendor-specific CNI integrations.
    - Kubernetes sees these as extra network resources; CNFs can be pinned to nodes with particular acceleration capabilities.
- **Operational considerations**
    - Node pools dedicated to GPU workloads; auto-scaling based on job queues or metrics.
    - Monitoring GPU utilization and allocations to avoid expensive idle capacity.

***

### Telco Networking: InfiniBand / RoCE for AI/ML clusters

- **Why low-latency, high-throughput interconnects**
    - Large-scale AI training and some telco analytics workloads require very fast node-to-node communication.
    - Traditional TCP/IP over Ethernet may become a bottleneck at scale.
- **InfiniBand and RoCE (RDMA over Converged Ethernet)**
    - Provide **Remote Direct Memory Access (RDMA)**, allowing data transfer directly between memory of different nodes with low CPU overhead.
    - Used in HPC and large AI clusters to speed up distributed training and data-plane operations.
- **Kubernetes integration patterns**
    - Use specialized CNI plugins and device plugins to expose RDMA-capable interfaces to Pods.
    - Pods request RDMA-capable network resources, and scheduler binds them to nodes wired to InfiniBand or RoCE fabrics.
- **Telco use cases**
    - AI-based RAN optimization, massive MIMO beamforming computations, or per-subscriber QoE analytics running on GPU/RDMA clusters.
    - CNFs that handle extremely high packet rates may leverage RDMA-enabled NICs.

## Wrap-up (10 min)

### Recap: Kubernetes as the "Operating System" of the datacenter

- **Kubernetes as an OS for clusters**
    - Like a traditional OS manages processes and resources on a single machine, Kubernetes manages **Pods and resources across many machines**.[^7][^3][^2]
    - Provides APIs for scheduling, networking, storage, and security; applications “ask” for what they need and Kubernetes decides where/how to run them.
- **Key mental models to retain**
    - Control plane vs worker nodes; reconciliation loops; declarative desired state.
    - Pods, Services, Deployments, and StatefulSets as core primitives.
    - Single-cluster vs multi-cluster layouts across VPCs, AZs, and regions for HA/DR and data sovereignty.
- **From infrastructure to platform**
    - Kubernetes plus surrounding components (GitOps, observability, security) turn raw compute into a **platform** that CNFs and applications can rely on.
    - Telco workloads (PNFs→VNFs→CNFs) increasingly assume such a platform exists and build on its capabilities.
- **What students should now be able to reason about**
    - How a YAML definition becomes real running containers on nodes.
    - How Kubernetes keeps applications healthy and reachable under failures.
    - How architecture choices (single vs multi-cluster, edge vs core, GPU/DPUs) impact latency, reliability, and compliance in modern telco environments.
