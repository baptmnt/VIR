# Lecture 3 – SRE, Security \& Practical Operations

**Objective:** Managing the lifecycle, reliability, and security of modern platforms in an “unhappy path” world.

***

## Introduction (10 min)

### Transitioning from "fixing servers" to engineering systemic reliability

- **Ops as engineering, not “heroic firefighting”**
    - Traditional ops: log into boxes, restart services, apply ad-hoc fixes; knowledge stays in people’s heads.
    - SRE-style ops: define **processes, automation, and metrics** so systems self-heal where possible and failures are predictable.[^1]
- **Why reliability is part of design, not an afterthought**
    - Reliability is influenced by architecture (stateless vs stateful, redundancy, deployment strategy) and release processes.
    - Bad deployments are a major cause of incidents; better pipelines + GitOps reduce risk.
- **Role of SRE for telco-like platforms**
    - Telco services must be up 24/7, so **releases, rollback, and monitoring** must be engineered carefully.
    - Even if students don’t become SREs, they will design or deploy services where SRE practices impact success.
- **Connection with previous lectures**
    - Lecture 1: how we moved to virtualized/containerized, multi-tenant infra.
    - Lecture 2: Kubernetes as the control plane.
    - Lecture 3: how we **operate and evolve** those platforms safely using CI/CD and GitOps.

## Automating Deployment (20 min)

### CI/CD \& GitOps: Git as "single source of truth" (Argo CD)

- **Continuous Integration (CI) – build \& test on every change**
    - CI systems (GitLab CI, GitHub Actions, Jenkins, etc.) build container images, run unit/integration tests, and scan for issues automatically on each commit or PR.
    - Outcome: a tested image in a registry and possibly generated Kubernetes manifests/Helm charts.
- **Continuous Delivery/Deployment (CD)**
    - Continuous Delivery: system is always in a **deployable** state; actual release may need human approval.
    - Continuous Deployment: passing CI results are automatically deployed to production (with safeguards like canaries).
- **GitOps principles**
    - **Git as the source of truth** for both application and infrastructure manifests; cluster state should match what’s in Git.[^2][^3][^4]
    - Changes flow: developer updates code/manifests → PR review → merge to main → automation applies changes to clusters.
- **Argo CD as a Kubernetes-native GitOps operator**
    - Argo CD runs inside Kubernetes and **watches Git repos** for desired configuration (YAML, Helm, Kustomize, etc.).[^3][^4][^2]
    - It compares Git state to cluster state, then **synchronizes** (apply/patch) resources until they match, reporting sync and health status.[^2][^3]
- **Pull vs push deployment models**
    - Traditional CD: CI/CD server **pushes** changes into the cluster (e.g., `kubectl apply` from Jenkins).
    - GitOps with Argo CD: cluster **pulls** desired state from Git; CI only builds images and updates manifests (e.g., image tag bump).[^4][^2]
    - Pull model simplifies firewalling and security (cluster initiates outbound connection to Git).
- **A typical CI + GitOps flow for Kubernetes**
    - CI builds container image, runs tests, pushes image to registry.
    - CI updates Kubernetes manifests or Helm values (image tag), commits to Git.
    - Argo CD detects repo change, syncs to cluster, then Kubernetes controllers perform rolling update.[^3][^4][^2]

***

### Strategies: Blue/Green \& Canary deployments

- **Why deployment strategies matter**
    - Most outages are caused by changes; strategies reduce blast radius and allow safe rollback.
    - Kubernetes + GitOps make these strategies easier to automate.[^5][^6][^7]
- **Blue/Green deployment**
    - Two identical environments: **Blue = current production**, **Green = new version**.[^6][^7][^5]
    - Process: deploy new version to Green, test it; when ready, switch traffic from Blue to Green (via load balancer, Ingress, or DNS change).
    - Advantages: instant rollback (switch back to Blue), clear separation; disadvantages: requires double capacity during rollout.[^5][^6]
- **Canary deployment**
    - Gradually route a **small percentage of traffic** to the new version while most users still hit the old version.[^7][^6][^5]
    - Observability (metrics, logs, traces) used to detect issues early; if problems appear, stop or rollback the canary.
    - More resource-efficient than blue/green, but requires the system to handle mixed versions concurrently.
- **Rolling update (Kubernetes default)**
    - Deployment controller updates Pods incrementally: terminate some old Pods, start some new Pods, respecting `maxSurge` and `maxUnavailable`.[^8][^7]
    - Gives zero-downtime updates for many stateless services but less fine-grained traffic shaping than canary.
- **Implementing strategies with Kubernetes + Argo CD**
    - Blue/Green: represent Blue and Green as different Deployments/Ingress backends; switch weights or service selector in Git; Argo CD syncs.[^6][^5]
    - Canary: use progressive delivery tools (Argo Rollouts, Flagger) that integrate with Service Mesh/Ingress to gradually shift traffic based on metrics.
    - GitOps: deployment strategy configuration (weights, labels, rollout spec) lives in Git, so every change is auditable.[^5][^2][^3]
- **Telco example**
    - Rolling out a new CNF version (e.g., signaling control microservice) first to 1% of network slices; monitor error rate and latency; then expand gradually if SLOs remain good.

## Observability \& Monitoring (20 min)

### Deep Observability: metrics, logs, and traces

- **Observability vs simple monitoring**
    - Monitoring: “is it up?”—basic CPU/memory checks, uptime graphs.
    - Observability: “**why** is it failing?”—correlated metrics, logs, and traces that allow understanding novel failure modes.[^1]
- **Metrics**
    - Numeric time series (e.g., request rate, latency, error rate, resource usage).
    - Exported by apps (Prometheus format) and infrastructure (Kubelet, cAdvisor, CNIs); used for dashboards and alerts (Prometheus, Grafana).
- **Logs**
    - Detailed event and error messages from apps and platform components.
    - Centralized log collection (Fluent Bit/Fluentd/Vector + Elasticsearch/OpenSearch/Loki) enables search across pods, nodes, and clusters.
- **Traces**
    - End-to-end request flows across microservices, showing spans, timing, and context (e.g., OpenTelemetry + Jaeger/Tempo).
    - Critical in telco-style microservice architectures where a call/session may traverse many services.
- **Observability in support of deployment strategies**
    - Canary/blue-green decisions based on metrics like error rate, P95 latency, or dropped requests; pipelines may automatically halt rollout when metrics go bad.[^9][^7][^5]
    - Argo Rollouts and similar tools can integrate metrics checks into rollout steps (progressive delivery).
- **Under the hood with Kubernetes**
    - kubelet and the control plane expose metrics on internal health (node conditions, pod restarts, queue lengths).
    - Labels and annotations on Pods/Namespaces make it possible to segment observability by service, team, or tenant.

## The SRE Framework (20 min)

### SLIs, SLOs, and SLAs: measuring health vs contracts

- **Service Level Indicator (SLI)**
    - A **quantitative metric** capturing user experience: e.g., “percentage of successful HTTP responses”, “median call setup time”, “P99 latency of control-plane API”.[^10][^9][^1]
    - Derived from observability data (metrics/logs/traces).
- **Service Level Objective (SLO)**
    - Target for an SLI over a period, e.g., “99.9% of requests succeed in a rolling 30-day window”.[^11][^10]
    - Set jointly by product, engineering, and ops; balances user expectations and engineering cost.
- **Service Level Agreement (SLA)**
    - Formal contract with customers, often with penalties (credits) if not met.
    - Usually less strict than internal SLOs; SLOs are used to keep SLA risk low.[^12][^1]
- **Error budget**
    - Allowed unreliability = 100% − SLO (e.g., SLO 99.9% → error budget 0.1% of time or requests).[^11][^10][^12]
    - Error budget consumption is tracked; if consumed too quickly, teams slow releases or prioritize reliability work.[^13][^11]
- **How this ties into deployment and GitOps**
    - CI/CD pipelines or deployment tools can use **error-budget-aware gates**: block or slow down releases when SLOs are not met.[^9][^12]
    - GitOps gives full history of configuration changes; when SLOs degrade, it’s easy to correlate incidents with recent config or release changes.

***

### Error Budgets: balancing speed and stability

- **Why 100% reliability is not the goal**
    - Chasing 100% is extremely expensive; small, controlled failures are acceptable if they enable faster innovation.[^12][^11][^1]
    - Error budget quantifies how much failure is acceptable in a period.
- **Using error budgets operationally**
    - When within budget: teams can deploy frequently and take more risk.
    - When budget is exhausted or close: freeze new releases, focus on fixing reliability issues (bugs, scaling limits, bad configs).[^10][^13][^11]
- **Error-budget policies**
    - Written rules specifying what to do when budget is exhausted (e.g., automatic change freeze, extra reviews, mandatory postmortems).[^13][^11]
    - Gives clear, pre-agreed decision framework rather than arguments during incidents.
- **Connection to deployment strategies**
    - Canary and blue/green deployments help **spend less error budget** per change by limiting blast radius.
    - If recent canaries consumed most of the error budget, the team might decide to pause further changes until metrics stabilize.
- **What students should retain**
    - SLIs/SLOs/error budgets are **guiding instruments** for how aggressive or conservative you can be with deployments.
    - They bridge the world of “reliability” with the world of **CI/CD cadence and business pressure**.

## Chaos Engineering: Failure Simulation (20 min)

### Beyond the "Happy Path": exploring via failure

- **Why chaos engineering exists**
    - Complex distributed systems fail in non-obvious ways; you only really know the system behavior by observing it under failure.
    - Chaos engineering introduces **controlled failures** (like dropping traffic, killing pods/nodes) to learn and improve resilience.
- **Principles of chaos experiments**
    - Start with a hypothesis: “If we kill one node, the service remains available.”
    - Run small, scoped experiments in non-production first; then possibly in production when mature.
- **Kubernetes-focused chaos tools**
    - Tools (e.g., Chaos Mesh, Litmus) can kill Pods, inject latency, corrupt network links, or simulate resource pressure at node level.
    - Experiments validate that Deployments, HPA, Pod disruption budgets, and self-healing behaviors work as expected.
- **Link with SRE and SLOs**
    - Chaos experiments verify that SLOs are still met under planned failure conditions.
    - Results feed back into architecture decisions (more replicas, better timeouts, enhanced autoscaling).

***

### Break-Fix Scenarios: diagnosing network and resource issues

- **Practicing “unhappy path” debugging**
    - Break-fix exercises intentionally reproduce issues: high latency, packet drops, CPU saturation, memory leaks.
    - Students learn to interpret metrics/logs and correlate symptoms (e.g., increased 5xx rate) with causes (e.g., new rollout, node failure).
- **Kubernetes-specific break-fix patterns**
    - Pod CrashLoopBackOff → investigate image, config, secrets, readiness/liveness probes.
    - Service unreachable → inspect endpoints, kube-proxy, NetworkPolicy rules.
    - Node under pressure → check resource requests/limits, scheduling, and noisy neighbors.
- **Telco-relevant scenarios**
    - Simulate partial loss of a cluster (zone outage) and see how services failover.
    - Simulate network latency between edge and core clusters and see impact on signaling or telemetry flows.
- **How CI/CD and GitOps help in break-fix**
    - Git history reveals exact infra and app changes before a failure.
    - Rollback is just a `git revert` or a sync to a previous commit; Argo CD handles the actual resources.[^4][^2][^3]

## Cloud Security Fundamentals (10 min)

### Shift Left: integrating security into the lifecycle

- **Shift-left security**
    - Move security checks earlier: in **code, CI, and Git** rather than only at runtime or perimeter.[^1]
    - Detect vulnerabilities and misconfigurations before they hit production.
- **IAM and least privilege**
    - Identity and Access Management (IAM/RBAC) defines who can change what: developers, pipelines, Argo CD, operators.
    - GitOps: Argo CD is a single, well-audited actor that applies changes; users mostly change Git, not the cluster directly.[^2][^3][^4]
- **Image scanning and supply chain security**
    - CI pipelines scan container images for known CVEs and disallow promotion of vulnerable images.
    - Manifests can be signed (Sigstore, Cosign) and verified before deployment.
- **Policy as code**
    - Tools like OPA/Gatekeeper enforce security policies on Kubernetes manifests (e.g., no privileged containers, required labels).
    - Argo CD can be configured to reject or flag out-of-policy configurations during sync.
- **Relevance for students**
    - Security is not just “firewalls”; in cloud-native, any deployment pipeline or misconfigured YAML can be an attack vector.
    - Understanding IAM, image scanning, and basic policy-as-code is essential for safe CI/CD and GitOps practices.

## Final Summary (10 min)

### The "T-shaped" professional: depth in containers, breadth in ops \& economics

- **T-shaped skill profile**
    - Vertical bar: **deep expertise** in containers, Kubernetes, and cloud-native design.
    - Horizontal bar: broad understanding of SRE concepts, deployment strategies, observability, security, and business trade-offs (cost vs reliability vs speed).
- **Connecting all three lectures**
    - Lecture 1: virtualized foundations, containers, and architectures that make cloud-native possible.
    - Lecture 2: Kubernetes as the platform orchestrating workloads across clusters and edges.
    - Lecture 3: CI/CD, GitOps, SRE, and security practices that keep that platform **reliable and evolvable** over time.
- **Key practical takeaways**
    - Design services with **deployability** in mind (health probes, statelessness where possible, configurable rollout strategies).
    - Use CI/CD and GitOps so that every infra and app change is traceable, reviewable, and reversible.[^3][^4][^2]
    - Measure what matters (SLIs), set reasonable SLOs, and let error budgets guide deployment velocity.[^11][^10][^9][^12]
- **Implication for telecom careers**
    - Future telecom roles blend networking knowledge with cloud, automation, and reliability engineering.
    - Even if they do not become SREs, understanding these mechanisms helps them collaborate effectively with SRE and platform teams.
