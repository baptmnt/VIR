# Lecture 1 – Virtualization \& Software Evolution

**Objective:** Understand the shift from rigid physical hardware to flexible, software-defined resources.

## Introduction (10 min)

### Scope: Transitioning from hardware constraints to "Cloud Native" agility

- **From hardware-defined to software-defined systems**
    - Historically, capabilities were tied to specific hardware boxes (routers, firewalls, HLRs, etc.).
    - Today, the same functions run as software on generic servers, VMs, or containers, in datacenters or clouds.
- **Rigid capacity vs elastic capacity**
    - Old world: capacity = number and size of physical boxes bought and installed. Long procurement cycles, overprovisioning “just in case”.
    - New world: capacity can scale up/down by starting or stopping VMs/containers and using autoscaling policies.
- **Manual operations vs programmable infrastructure**
    - Traditional ops: log into boxes, run commands by hand, apply vendor-specific procedures.
    - Cloud-native ops: APIs and code (Terraform, Ansible, Kubernetes) define and manage infra; repeatable and automated.
- **Why this matters for telecom students**
    - Telecom networks are becoming software-centric (5G core, IMS, SD-WAN).
    - Understanding virtualization and cloud-native patterns is essential to design, operate, and optimize modern telco platforms, not just IP/MPLS or RF.

***

### Telco Hook: Moving from physical network appliances (PNFs) to virtualized functions (VNFs/CNFs)

- **PNFs – Physical Network Functions**
    - Hardware appliances implementing network functions: routers, firewalls, SBCs, EPC nodes, etc.
    - Tight coupling of hardware and software; each box is “one purpose”; scaling requires buying/installing more boxes.[^1][^2]
- **VNFs – Virtualized Network Functions**
    - Network functions packaged as software running on **virtual machines** on COTS servers, using hypervisors (VMware, KVM, OpenStack).[^3][^2][^4]
    - Decouples function from dedicated hardware: same server can run multiple VNFs; easier scaling and lifecycle management.
- **CNFs – Cloud-/Container-Native Network Functions**
    - Network functions decomposed into **microservices** running in **containers**, orchestrated by Kubernetes.[^5][^4]
    - Designed for elasticity, automation (CI/CD), and operation across many clusters and locations (core, edge).
- **Why PNFs → VNFs → CNFs is more than a buzzword chain**
    - Reflects deeper evolution: from hardware thinking, to VM thinking, to cloud-native thinking (resilience, automation, observability).
    - This lecture explains the generic computing concepts that underpin this telco evolution so students can generalize beyond any specific vendor.

## The Transition to Virtualization (20 min)

### The Bare Metal Era: "One Server, One App"

- **Direct deployment on the OS (no virtualization)**
    - Applications installed directly on a physical server’s OS (often Linux/Unix/Windows).
    - The OS manages processes, memory, storage, and network directly on hardware.
- **The “One Server, One App” practice**
    - To avoid conflicts and performance unpredictability, admins dedicated each physical server to a single major application (e.g., one billing app, one HLR front-end, one NMS server).
    - This reduced “dependency hell” but wasted resources: CPU and memory mostly idle.
- **Underutilization and cost**
    - Average CPU utilization often <20–30% because capacity was reserved for peak loads that arrived rarely.
    - For telcos, this means large CAPEX and OPEX for hardware, power, and datacenter space.
- **Dependency hell and “snowflake” servers**
    - Many apps required specific OS versions, libraries, kernel modules; upgrading one app risked breaking another.
    - Manual installs created unique, fragile configurations (“snowflake servers”) that were hard to reproduce or recover.
- **Operational pain points**
    - Long provisioning cycles: order hardware → rack/stack → install OS → configure app → weeks/months to deliver a service.
    - Failures were painful: if a box died, service died until that specific machine was fixed (no easy replacement).

***

### The Hypervisor Layer: Hardware abstraction with VMware/KVM

- **Concept of a hypervisor**
    - Software layer that **emulates virtual hardware** and runs multiple **virtual machines (VMs)** on one physical server.[^3]
    - Each VM runs its own **Guest OS** (Linux, Windows, etc.) as if it were on dedicated hardware.
- **Types of hypervisors (high-level)**
    - Type 1 (bare-metal): ESXi, Hyper-V, KVM on bare metal; typically used in datacenters.
    - Type 2 (hosted): VirtualBox, VMware Workstation; mainly for desktops/labs.
- **Benefits introduced by virtualization**
    - **Consolidation:** multiple workloads share a server, dramatically improving utilization.
    - **Isolation:** one VM crashing or misbehaving does not directly affect others; better security boundaries.
    - **Abstraction from hardware:** VMs can be migrated to other hosts (vMotion/Live Migration) with no app change.
- **Impact on provisioning and operations**
    - Provisioning a new VM becomes a **software action** (API/click), not a hardware purchase.
    - Snapshots, cloning, and templating of VMs make environments faster to replicate for testing/staging.
    - Disaster recovery: easier to replicate VM images across sites.
- **Telco implications**
    - VNFs become possible: firewalls, EPC, IMS nodes running inside VMs on COTS servers instead of custom appliances.[^2][^4]
    - NFV stacks (e.g., ETSI NFV-MANO) orchestrate VNFs on hypervisor-based infrastructures, paving the way to CNFs.

## Modern Software Architectures (25 min)

### Microservices: From Monolith to small, independent, networked units

- **The monolith model**
    - One large codebase, one deployable unit, often one database schema.
    - Easy to start, but hard to evolve: releases are “big bang”; small changes require full redeploy.
- **What are microservices?**
    - Application split into **small, independently deployable services**, each owning a specific business capability (e.g., “billing”, “customer profile”, “session management”).
    - Communicate via lightweight protocols (HTTP/REST, gRPC, message queues).
- **Benefits of microservices**
    - **Independent scaling:** scale only the services under high load (e.g., call control vs. admin GUI).
    - **Independent deployment:** teams release their components at different cadences without waiting for a full release train.
    - **Technology diversity:** each service can use its own tech stack if interfaces are stable.
- **Challenges and trade-offs**
    - Increased **operational complexity**: many services to deploy, monitor, and debug (where does a failure occur?).
    - Network becomes part of application design (timeouts, retries, backpressure, observability).
    - Requires strong automation (CI/CD) and mature runtime (orchestrators, service meshes).
- **Telco relevance**
    - 5G Core and modern network control planes are designed as microservices to support **network slicing**, dynamic scaling, and rapid feature rollout.
    - CNFs are often implemented as sets of collaborating microservices.

***

### Hexagonal Architecture: Decoupling business logic from infrastructure

- **Core idea of Hexagonal (Ports \& Adapters) architecture**
    - Place **business logic at the center**, isolated from details like databases, UIs, messaging systems, or external APIs.[^6][^7]
    - Communication with the outside world happens via **ports** (interfaces) and **adapters** (implementations).
- **Ports and Adapters explained**
    - **Ports:** abstract interfaces that describe **what** the application needs or provides (e.g., “StoreSubscriberProfile”, “SendSMSNotification”).
    - **Adapters:** concrete code that connects ports to technologies: SQL databases, Kafka, REST APIs, CLI, etc.[^8][^6]
    - Swapping adapters (e.g., change DB or message bus) should not affect domain/business logic.
- **Why this matters for virtualization and cloud-native**
    - If business code is decoupled from infrastructure, the same logic can run on bare metal, VM, or container **without changes**.
    - Facilitates migration from PNF software → VNF → CNF by minimizing hard-coded infrastructure dependencies.
- **Testability and resilience**
    - With hexagonal architecture, domain logic can be tested in isolation using fake adapters (in-memory DB, mock APIs).
    - Easier to introduce feature flags, retry logic, timeouts at the adapter level, improving reliability under failure.
- **Telco examples**
    - A subscriber management service whose core logic is independent of whether it stores data in an on-prem Oracle DB, a cloud-native PostgreSQL, or a distributed key-value store.
    - Same call-routing logic used whether signaling arrives via SIP over UDP or HTTP/2, by plugging in appropriate adapters.

## Scaling \& Isolation (20 min)

### Vertical vs. Horizontal: "Bigger Boxes" vs. "More Boxes"

- **Vertical scaling (“scale up”)**
    - Add more resources to a single machine/VM: more CPU cores, RAM, faster disks.
    - Limited by physical maximums and often costly hardware upgrades.
- **Horizontal scaling (“scale out”)**
    - Add more instances of your service (VMs/containers) behind a load balancer.
    - Natural fit for microservices and stateless workloads; aligns with the “cattle” mindset.
- **Performance vs resilience implications**
    - Vertical scaling can boost performance but creates **bigger single points of failure**.
    - Horizontal scaling enables **fault tolerance**: if one instance fails, others keep serving traffic.
- **Role of virtualization in scaling**
    - Hypervisors allow quick instantiation of more VMs; containers make it even cheaper and faster to add instances.
    - Orchestrators (Kubernetes, OpenStack autoscaling, NFV orchestrators) automate scaling decisions.
- **Telco use cases**
    - Handling busy hour or special events (sports, concerts) by scaling out call-processing or streaming services.
    - Scaling signaling/control-plane services separately from data-plane functions.

***

### Multi-Tenancy: How virtualization enables cost-effective, instant horizontal scaling

- **What is multi-tenancy?**
    - Multiple **tenants** (customers, teams, or products) share the same underlying infrastructure and sometimes the same software instance, while keeping their data and traffic isolated.[^9][^10]
    - Opposed to single tenancy, where each tenant has its own dedicated stack.
- **Multi-tenancy vs “just VMs”**
    - Virtualization isolates OS instances per tenant but may still allocate dedicated VMs per customer.
    - Application-level multi-tenancy lets one app/DB instance serve many tenants securely within the same environment.[^11][^9]
- **Benefits of multi-tenancy**
    - **Cost efficiency:** shared hardware and operations lower per-tenant cost.
    - **Elastic resource pooling:** CPU/RAM/storage pooled and dynamically allocated; idle capacity of one tenant can serve others.
    - **Simplified operations:** centralized upgrades, patching, and monitoring across many tenants.
- **Isolation mechanisms enabling multi-tenancy**
    - At infra level: hypervisor boundaries, container isolation (namespaces, cgroups), network segmentation (VLANs/VXLANs, VRFs).
    - At app level: tenant IDs, per-tenant encryption keys, access-control layers.
- **Telco-specific examples**
    - MVNOs or enterprise customers sharing the same 5G core instance as logical slices.
    - Private networks hosted as tenants on an operator’s shared cloud or edge infrastructure.

## Hypervisors vs. Containers (25 min)

### Virtual Machines: Full hardware abstraction; each has its own Guest OS

- **VM structure**
    - Physical server → hypervisor → multiple VMs → each VM with its own OS kernel, libraries, and applications.
    - Hypervisor intercepts privileged operations and provides virtualized CPU, memory, storage, and NICs.
- **Advantages of VMs**
    - **Strong isolation:** separate OS per VM; good security boundary between tenants.
    - **Compatibility:** can run different OSes on the same host (Linux and Windows side by side).
    - **Mature ecosystem:** tools for backup, live migration, snapshots, and high availability are well established.
- **Drawbacks of VMs**
    - Heavier resource footprint: each VM includes its own OS kernel and system services.
    - Slower provisioning compared to containers; less density on the same hardware.
    - Overhead in boot time and image size.
- **When VMs are still the right tool**
    - Strict isolation requirements, legacy workloads, different OS families, or when hypervisor-level features (e.g. SR-IOV for NICs) are critical.
    - Many VNFs are still delivered as VM-based appliances, especially in transitional or hybrid environments.[^4][^5]

***

### Containers: OS-level virtualization using Linux primitives (namespaces \& cgroups)

- **What are containers?**
    - Processes on a Linux host that are **isolated by the kernel** using namespaces and **resource-limited** via cgroups.[^12][^13]
    - Share the host kernel but have their own filesystem, process tree, and network stack view.
- **Core Linux primitives**
    - **Namespaces:** provide isolated views of resources (PID, NET, MNT, IPC, UTS, USER, CGROUP).[^14][^15]
        - PID: each container sees only its own processes.
        - NET: virtual network stack per container (interfaces, routing tables).
        - MNT: isolated filesystem hierarchy.
        - USER: different user IDs inside vs. outside, improves security.
    - **cgroups (control groups):** limit and account for CPU, memory, I/O, etc., preventing noisy neighbors in multi-tenant environments.[^16][^13][^12]
- **How containers differ from VMs**
    - Containers **do not include a full OS kernel**; they rely on the host’s kernel.
    - Much lighter images and faster startup (milliseconds to seconds).
    - Higher density: many more containers than VMs on the same hardware, ideal for microservices and CNFs.
- **Security and isolation considerations**
    - Container isolation is strong but generally considered less strict than VM isolation; hardening is needed (seccomp, AppArmor/SELinux, rootless containers).[^15][^14]
    - For sensitive multi-tenant telco workloads, combinations like microVMs (Firecracker) or Kata Containers can provide stronger isolation with container workflows.
- **Why containers are foundational for cloud native \& telco CNFs**
    - Align perfectly with microservices: each microservice runs as one or few containers.
    - Kubernetes and similar platforms assume containers as deployment units, enabling autoscaling, self-healing, and service discovery.
    - CNFs use containers to pack more network functions per host and scale them rapidly in response to traffic changes.[^5][^4]

## The Deployment Revolution (10 min)

### Infrastructure as Code (IaC): Treating infra like software (Terraform/Ansible)

- **What is Infrastructure as Code?**
    - Managing and provisioning infrastructure **via code** (declarative configuration files) rather than manual CLI/GUI steps.[^17]
    - Infra definitions are version-controlled, code-reviewed, and tested like application code.
- **Terraform vs. Ansible (high-level roles)**
    - **Terraform:** declarative, state-based tool for creating and changing cloud and virtualization resources (VMs, networks, load balancers, managed services).[^18][^17]
    - **Ansible:** procedural configuration management and automation for installing packages, configuring services, deploying applications on those resources.
    - Typical workflow: Terraform provisions infrastructure → Ansible configures and deploys the software stack.[^19][^17]
- **Why IaC is critical in a virtualized/multi-tenant world**
    - With VMs and containers, environments are dynamic and ephemeral; manual configuration does not scale.
    - IaC ensures **repeatability**: dev, test, and prod environments are reproducible from code.
    - Enables rapid recovery: if a cluster fails, re-create it from IaC definitions instead of troubleshooting “by hand”.
- **Benefits for operations and governance**
    - **Auditability and compliance:** config changes tracked in Git history; easier to prove what is running where.
    - **Collaboration:** infra defined in shared repos; developers, ops, and security can contribute.
    - **Standardization:** reusable modules and playbooks enforce best practices across many teams and tenants.
- **Telco angle**
    - NFV/CNF environments often rely on IaC to provision NFVI (compute, storage, network) and deploy VNFs/CNFs consistently across many sites and regions.
    - IaC pipelines are key to rolling out upgrades and patches across hundreds of telco edge sites without manual intervention.

## Wrap-up (10 min)

### Recap: The evolution to "Cattle, not Pets"

- **Pets vs Cattle metaphor**
    - **Pets:** individually named servers (e.g., “router-paris-01”), manually maintained, irreplaceable; if they fail, everyone panics.[^20][^21][^22]
    - **Cattle:** fungible instances identified by numbers (vm-001, pod-xyz); if one fails, it is replaced automatically by orchestration, not “repaired by hand”.[^23][^24]
- **How each stage of evolution supports the “cattle” mindset**
    - Bare metal era: mostly “pets”; configuration drift and manual tuning dominate.
    - Hypervisors: easier cloning and migration of VMs; first step toward treating infra as replaceable units.
    - Containers + orchestrators: very fast, highly automated replacements; infrastructure finally behaves like “cattle”.
    - IaC: makes the **desired state** codified; any instance is just an implementation detail of that state.
- **Conceptual summary of the lecture**
    - **From hardware constraints to software-defined flexibility:** virtualization breaks the “one box, one role” limitation.
    - **From monoliths to microservices \& hexagonal architectures:** software is split into decoupled units that can be deployed and scaled independently, and are agnostic to underlying infra.
    - **From manual operations to IaC \& automation:** everything important is defined as code; infra is reproducible and disposable.
    - **From single-tenant boxes to multi-tenant platforms:** shared infra with strong isolation and dynamic resource allocation is the default model.
- **Why this matters for your careers as telecom engineers**
    - Future networks are **programmable, virtualized, and cloud-native**; understanding these concepts is as important as understanding routing protocols or radio access.
    - Being able to think in terms of **platforms, automation, and reliability** will differentiate engineers who design the next generation of telco services from those who only operate legacy equipment.
